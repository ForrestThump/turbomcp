# turbomcp-dpop Migration Guide

For workspace-wide breaking changes (unified errors, transport crate splits, MCP spec updates),
see the [top-level MIGRATION.md](../../MIGRATION.md).

---

## v2.x to v3.0

### Dependency update

```toml
[dependencies]
turbomcp-dpop = "3.1.2"

# Or via the umbrella crate with auth feature:
turbomcp = { version = "3.1.2", features = ["auth", "dpop"] }
```

### RSA algorithm removal

v3.0 removes `DpopAlgorithm::RS256` and `DpopAlgorithm::PS256` to eliminate timing-attack
vulnerabilities (RUSTSEC-2023-0071). Only ES256 (ECDSA P-256) is supported.

```rust
// Before (v2.x) - RS256 variant existed on DpopAlgorithm
// (e.g. via DpopKeyManager::generate_key_pair(DpopAlgorithm::RS256))

// After (v3.0) - ES256 only; direct helper is generate_p256()
let key_pair = DpopKeyPair::generate_p256()?;

// Or via DpopKeyManager (the only remaining algorithm is ES256):
// let manager = DpopKeyManager::new_memory().await?;
// let key_pair = manager.generate_key_pair(DpopAlgorithm::ES256).await?;
```

Any `match` arms or configuration values referencing the removed variants will produce compile
errors. Update them to `DpopAlgorithm::ES256`.

### Unified error types

In v2.x the crate exposed a mix of local and upstream error types. In v3.0 `DpopError` is the
single error type for all DPoP operations, and `turbomcp_dpop::Result<T>` is defined as
`Result<T, DpopError>`.

```rust
use turbomcp_dpop::{DpopError, Result as DpopResult};
```

### `DpopProofParams` and `DpopProofParamsBuilder`

Proof construction uses `DpopProofParams` (built via `DpopProofParamsBuilder`, generated by the
`bon` derive macro). These types are re-exported at the crate root.

```rust
use turbomcp_dpop::{DpopProofParams, DpopProofParamsBuilder};
```

### `DpopValidator` and `ValidatedDpopClaims`

Proof validation uses `DpopValidator` which produces `ValidatedDpopClaims` on success. Both are
re-exported at the crate root.

```rust
use turbomcp_dpop::{DpopValidator, ValidatedDpopClaims};
```

## v1.x to v2.0

### New standalone crate

DPoP types lived inside the main `turbomcp` crate in v1.x. In v2.0 they were extracted into
the dedicated `turbomcp-dpop` crate.

```toml
# Before (v1.x)
[dependencies]
turbomcp = { version = "1.x", features = ["dpop"] }

# After (v2.0) - dedicated crate
[dependencies]
turbomcp-dpop = "2.0"

# Or continue using the umbrella crate (re-exports are preserved)
turbomcp = { version = "2.0", features = ["auth", "dpop"] }
```

### Import paths

```rust
// Before (v1.x)
use turbomcp::dpop::DpopProofParams;

// After (v2.0+) - direct crate
use turbomcp_dpop::DpopProofParams;

// After (v2.0+) - via umbrella crate
use turbomcp::dpop::DpopProofParams;
```

### Correct type names

The table below lists the types that actually exist in the codebase. Earlier documentation
contained incorrect names; do not use the names in the "does not exist" column.

| Does not exist (incorrect) | Correct name | Location |
|----------------------------|--------------|----------|
| `DPopProofBuilder` | `DpopProofParams` / `DpopProofParamsBuilder` | `turbomcp_dpop::helpers`, re-exported at crate root |
| `RedisNonceStore` | `RedisNonceStorage` | `turbomcp_dpop::redis_storage` (feature: `redis-storage`) |
| `YubiHsmKeyStore` | `YubiHsmManager` | `turbomcp_dpop::hsm::yubihsm` (feature: `hsm-yubico`) |

Additional types re-exported at the crate root:

| Type | Source module | Notes |
|------|---------------|-------|
| `DpopAlgorithm` | `types` | Algorithm enum; only `ES256` in v3.0+ |
| `DpopKeyPair` | `types` | Key pair for signing proofs |
| `DpopPrivateKey` | `types` | Private key enum |
| `DpopPublicKey` | `types` | Public key enum |
| `DpopKeyMetadata` | `types` | Key metadata and rotation info |
| `DpopProof` | `types` | Validated proof structure |
| `DpopError` | `errors` | Unified error type |
| `DpopValidator` | `helpers` | Proof validation |
| `ValidatedDpopClaims` | `helpers` | Output of successful validation |

HSM types (not re-exported at crate root; import from submodules):

| Type | Module | Feature |
|------|--------|---------|
| `HsmConfig` | `turbomcp_dpop::hsm` | any HSM feature |
| `Pkcs11Config` / `Pkcs11ConfigBuilder` | `turbomcp_dpop::hsm` | `hsm-pkcs11` |
| `YubiHsmConfig` / `YubiHsmConfigBuilder` | `turbomcp_dpop::hsm` | `hsm-yubico` |
| `YubiHsmConnector` | `turbomcp_dpop::hsm` | `hsm-yubico` |
| `YubiHsmManager` | `turbomcp_dpop::hsm::yubihsm` | `hsm-yubico` |
| `Pkcs11HsmManager` | `turbomcp_dpop::hsm::pkcs11` | `hsm-pkcs11` |
| `HsmManager` | `turbomcp_dpop::hsm` | any HSM feature |

### Feature flags

Feature names on `turbomcp-dpop` are unchanged from v1.x to v2.0. Use the names below when
depending on the crate directly.

```toml
turbomcp-dpop = { version = "2.0", features = [
    "redis-storage",  # Redis nonce storage (RedisNonceStorage)
    "hsm-pkcs11",     # PKCS#11 HSM support
    "hsm-yubico",     # YubiHSM support
    "hsm",            # All HSM backends (hsm-pkcs11 + hsm-yubico)
    "test-utils",     # Test utilities
] }
```

### Redis nonce storage

```rust
// Before (v1.x) - incorrect type name used in old docs
use turbomcp::dpop::storage::RedisNonceStore;  // did not exist

// After (v2.0+) - correct type
use turbomcp_dpop::redis_storage::RedisNonceStorage;
```

Requires the `redis-storage` feature.

### YubiHSM integration

```rust
// Before (v1.x) - incorrect type name used in old docs
use turbomcp::dpop::hsm::YubiHsmKeyStore;  // did not exist

// After (v2.0+) - correct types
use turbomcp_dpop::hsm::{YubiHsmConfig, YubiHsmConfigBuilder, YubiHsmConnector};
use turbomcp_dpop::hsm::yubihsm::YubiHsmManager;
```

Requires the `hsm-yubico` feature.

### Proof construction

```rust
use turbomcp_dpop::{DpopProofParams, DpopKeyPair};

let key_pair = DpopKeyPair::generate_p256()?;

// DpopProofParams is constructed via its bon-generated builder.
// http_method and http_uri are String-typed (anything Into<String> works).
let proof = DpopProofParams::builder()
    .http_method("POST")
    .http_uri("https://api.example.com/token")
    .build()
    .build_with_key(&key_pair)
    .await?;
```
